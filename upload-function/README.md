# Artnet Upload Function

Pure storage service for new lot-image uploads.

This component coordinates uploads into Google Cloud Storage. It does not write to any application database, and it does not proxy the image bytes during the direct-upload flow. Callers receive the canonical `objectKey` and `publicUrl`, then write whatever database rows their own application requires.

## Current Status

Implemented:

- `GET /v1/health`
- `POST /v1/uploads/presign`

- R2 replication on finalize (spec §2.9)

- browser CORS/preflight handling
- Redis-backed upload sessions, with an explicit in-memory fallback for tests/local-without-Docker
- runnable browser and server examples

Not implemented yet:

- Reconciliation Function.

Those deferred items are separate stages in `docs/spec.md` section 10.

## Direct Upload Flow

The direct-upload flow has three steps:

```text
caller -> API gateway -> Upload Function: POST /v1/uploads/presign
caller -> GCS signed URL:                 PUT image bytes
caller -> API gateway -> Upload Function: POST /v1/uploads/:uploadId/finalize
```

Only the control-plane calls go through Artnet infrastructure. The large image upload goes directly to GCS using a short-lived signed URL.

### Why Signed URLs?

The signed URL is a temporary bearer credential generated by the Upload Function. It is scoped to:

- one private staging object key
- one method, `PUT`
- one expiration window, default `900` seconds

This avoids sending 50 MB files through the API gateway or Cloud Run Function.

## Object Keys

`objectKey` is the image's durable internal storage address. It is used consistently by GCS, future R2 replication, reconciliation, and callers that persist upload metadata.

Direct-upload signed URLs do not write to `objectKey` immediately. They write to a private staging key:

```text
staging/uploads/<uploadId>/<objectKey>
```

`finalize` validates the staged object, promotes it to the canonical `objectKey`, deletes the staged object, then replicates the canonical object to R2. Abandoned staged objects are deleted by the GCS lifecycle rule after 5 days.

New direct uploads use product-prefixed keys:

```text
products/galleries/artworks/<galleryId>/<artworkId>/images/<imageId>[suffix].<ext>
products/artnet-auctions/auction-lots/<auctionHouseId>/<auctionDate>/<lotId>/images/<imageId>[suffix].<ext>
products/pdb/artworks/<pdbArtworkId>/images/<imageId>[suffix].<ext>
```

Legacy unprefixed auction keys remain valid for existing images:

```text
lot_images/425939177/20260310/638775/195.jpg
```

Examples:

```text
products/galleries/artworks/gallery-1/artwork-1/images/195.jpg
products/artnet-auctions/auction-lots/425939177/20260310/638775/images/195.jpg
products/pdb/artworks/pdb-123/images/195.jpg
```

The extension is derived from `contentType`:

- `image/jpeg` -> `.jpg`
- `image/png` -> `.png`
- `image/webp` -> `.webp`

## Upload Sessions

An upload session is short-lived state created during `presign` and consumed during `finalize`.

It records:

- `uploadId`
- `objectKey`
- staging object key
- `productId`
- `auctionHouseId` for auction-lot uploads
- expected `contentType`
- expected `contentLength`
- expiration time

During finalize, the function looks up the session, checks that the staged GCS object exists, verifies its size and content type, promotes it to the canonical key, returns metadata, and deletes the session.

The production implementation stores sessions in Redis through `ioredis`. Each session is written under `REDIS_KEY_PREFIX + uploadId` with a TTL matching the signed URL expiry. This lets presign and finalize land on different Cloud Run instances.

Set `SESSION_STORE=redis` for normal local and deployed runs. Use `SESSION_STORE=memory` only for tests or local development without Docker. In-memory sessions are lost if the process restarts and are not safe for horizontally scaled deployments.

The in-memory fallback also enforces a soft capacity ceiling (default `10_000` sessions). When the ceiling is reached, expired sessions are swept first; if the store is still full, new presigns are rejected with `503 too_many_sessions`.

## Cloud Run Deployment

Production Cloud Run deployments should configure `SESSION_STORE=redis` and `REDIS_URL` so session state is shared across instances. With Redis enabled, Cloud Run can scale horizontally and the spec's `--max-instances=10` setting is compatible with direct uploads.

Only local/dev deployments that intentionally omit Redis should cap the function to one instance:

```text
--min-instances=1
--max-instances=1
```

The function does not create or manage the Redis cluster. It expects the deployment environment to provide a reachable Redis URL, including TLS/auth details if required. In GCP lower/prod environments this points at Memorystore through a Serverless VPC Connector.

## Gateway Boundary

This implementation assumes authentication and admission control happen before requests reach the function, typically at an API gateway.

The function does not verify bearer tokens.

The gateway must inject product context on every `/v1/uploads/*` and `DELETE /v1/objects/*` request:

```text
X-Artnet-Product-Id: galleries | artnet-auctions | pdb
```

The header is required. Requests without it are rejected with `403 product_required`. The function enforces it against the request entity kind, upload session, or object key product. Mismatches return `403 product_mismatch`.

For `auction-lot` uploads and auction-lot object deletes, the gateway must also inject auction-house context:

```text
X-Artnet-Auction-House-Id: 425939177
```

The auction-house header is required only for auction-lot paths. Requests without it are rejected with `403 auction_house_required`. When present, the function enforces it against:

- the presign request `auctionHouseId`
- the upload session during finalize
- the upload session during cancel
- the object key during finalized object delete

Mismatches return `403 auction_house_mismatch`.

Important: browsers must not be trusted to supply `X-Artnet-Product-Id` or `X-Artnet-Auction-House-Id` directly. The gateway must strip client-provided values and replace them with its own trusted values derived from the verified session.

## Local Setup

Install dependencies:

```bash
npm install
```

Start local Redis for the normal dev path:

```bash

```

```bash

```

Fill in `.env`:

```bash
GCS_BUCKET=<your-dev-gcs-bucket>
PUBLIC_BASE_URL=https://artworks.artnet.com
SIGNED_URL_TTL_SECONDS=900
CORS_ALLOW_ORIGIN=*
R2_ACCOUNT_ID=<your-r2-account-id>
R2_BUCKET=<your-r2-bucket>
R2_ACCESS_KEY_ID=<your-r2-access-key-id>
R2_SECRET_ACCESS_KEY=<your-r2-secret-access-key>
S3_SOURCE_ENDPOINT=https://<account-or-provider-endpoint>
S3_SOURCE_REGION=auto
S3_SOURCE_ACCESS_KEY_ID=<your-source-access-key-id>
S3_SOURCE_SECRET_ACCESS_KEY=<your-source-secret-access-key>
S3_SOURCE_ALLOWED_BUCKETS=artnet-vendor-feed
SESSION_STORE=redis
REDIS_URL=redis://127.0.0.1:6379
REDIS_KEY_PREFIX=upload-session:
```

The direct-upload examples require a real dev GCS bucket because they upload to the signed GCS URL returned by `presign`.

`npm run dev` loads `.env` automatically using Node's `--env-file-if-exists=.env` flag. You do not need to export these variables manually.

## API Summary

### `GET /v1/health`

Checks that the configured GCS bucket, R2 bucket, S3 source bucket allowlist, and upload session store are reachable.

Response:

```json
{
  "status": "ok"
}
```

### `POST /v1/uploads/presign`

Creates an upload session and returns a signed GCS URL.

Request:

```json
{
  "kind": "auction-lot",
  "auctionHouseId": "425939177",
  "auctionDate": "20260310",
  "lotId": "638775",
  "imageId": "195",
  "imageVariantSuffix": null,
  "contentType": "image/jpeg",
  "contentLength": 1024576
}
```

Response:

```json
{
  "uploadId": "01JF8T7P2C9X4MWXYZ",
  "objectKey": "products/artnet-auctions/auction-lots/425939177/20260310/638775/images/195.jpg",
  "uploadUrl": "https://storage.googleapis.com/<signed-v4-url>",
  "uploadHeaders": {
    "Content-Type": "image/jpeg",
    "Content-Length": "1024576",
    "X-Goog-Content-Length-Range": "1024576,1024576"
  },
  "expiresAt": "2026-05-07T18:30:00.000Z"
}
```

### Signed URL `PUT`

Upload bytes directly to the returned `uploadUrl`. Every header in `uploadHeaders` is part of the v4 signature — callers must send each one with the exact value returned, or GCS will reject the PUT with `403 SignatureDoesNotMatch`.

Server-side callers send all three:

```text
Content-Type: <uploadHeaders.Content-Type>
Content-Length: <uploadHeaders.Content-Length>
X-Goog-Content-Length-Range: <uploadHeaders.X-Goog-Content-Length-Range>
```

Browser callers send `Content-Type` and `X-Goog-Content-Length-Range` and let the browser set `Content-Length`. The GCS bucket CORS policy must allow `X-Goog-Content-Length-Range` in `AllowedHeaders`; otherwise the preflight blocks the upload.

`X-Goog-Content-Length-Range` is set to `<contentLength>,<contentLength>` so GCS itself rejects any PUT whose body is not exactly the size declared at presign. This is defense in depth alongside the size check on finalize.

### `POST /v1/uploads/:uploadId/finalize`

Verifies the uploaded GCS object and returns canonical metadata.

Response:

```json
{
  "objectKey": "products/artnet-auctions/auction-lots/425939177/20260310/638775/images/195.jpg",
  "publicUrl": "https://artworks.artnet.com/_v/01ARZ3NDEKTSV4RRFFQ69G5FAV/products/artnet-auctions/auction-lots/425939177/20260310/638775/images/195.jpg",
  "size": 1024576,
  "contentType": "image/jpeg",
  "uploadedAt": "2026-05-07T18:15:32.412Z",
  "replicatedToR2": false
}
```

Finalize verifies staged object existence, size, and content type. It does not compute or return a direct-upload checksum. Objects rejected for size or content-type mismatch are deleted from staging in GCS. Valid staged objects are promoted to the canonical `objectKey` before R2 replication.

`publicUrl` includes `/_v/<uploadId>/` before the canonical `objectKey`. This keeps the storage key stable while giving browsers and Cloudflare a fresh immutable URL after every successful upload or re-upload.

`replicatedToR2` is `true` when the GCS object was successfully mirrored to R2 during finalize per spec §2.9. If R2 was unreachable after retries, finalize still returns 200 with `replicatedToR2: false` and the Reconciliation Function (§4) will repair the drift later.

### `DELETE /v1/uploads/:uploadId`

Cancels an in-flight upload session.

Response:

```text
204 No Content
```

If the session still exists, cancel also deletes the staged GCS object. If the caller abandons the upload without canceling, Redis expires the session after the signed URL TTL and GCS lifecycle deletes `staging/uploads/` objects after 5 days.

### `DELETE /v1/objects/<objectKey>`

Deletes a finalized image from GCS and R2. Spec §2.3.6.

The `<objectKey>` tail is the full canonical key, e.g.:

```text
DELETE /v1/objects/lot_images/425939177/20260310/638775/195.jpg
X-Artnet-Product-Id: artnet-auctions
X-Artnet-Auction-House-Id: 425939177
```

Product-prefixed keys are deleted the same way, for example:

```text
DELETE /v1/objects/products/galleries/artworks/gallery-1/artwork-1/images/195.jpg
X-Artnet-Product-Id: galleries
```

Behavior:

1. Validate trusted product context and the object key shape (spec §2.6). Enforce that `X-Artnet-Product-Id` matches the product encoded in the key. For auction-lot keys, also enforce that `X-Artnet-Auction-House-Id` matches the `auctionHouseId` segment in the path — otherwise `403 auction_house_mismatch`.
2. Write a tombstone (spec §2.12) at `tombstones/<objectKey>.json` in GCS. Reconciliation reads this to avoid backfilling a deleted object from GCS to R2.
3. Delete the object from R2 (retries per `R2_REPLICATION_RETRIES`; 404 treated as success).
4. Delete the object from GCS (404 treated as success).

Response: `204 No Content`. The endpoint is idempotent — repeat calls succeed silently.

Failure handling: each step retries transient errors internally. If a step ultimately fails, the function returns `503` and the caller retries. Retries are safe because the tombstone overwrite is harmless and both delete operations swallow 404.

Tombstone GCS lifecycle: deploy a lifecycle policy on the `tombstones/` prefix to expire markers after 7 days (well past the reconciler's 48-hour modified-objects window). Documented as a deployment concern; not enforced in code.

Staging GCS lifecycle: deploy a lifecycle policy on the `staging/uploads/` prefix to expire abandoned direct-upload objects after 5 days. Terraform in `infra/gcp` includes this rule.

```json
{
  "error": {
    "code": "upload_not_received",
```

## R2 Replication

R2 replication during finalize is implemented per spec §2.9.

After the GCS object is verified during finalize, the function streams it to R2 at the same `objectKey`. The S3-compatible client retries transient failures (configured via `R2_REPLICATION_RETRIES`, default 3). On success `replicatedToR2: true`. On exhausted retries the function logs `r2.replication.failed` and returns 200 with `replicatedToR2: false`; the Reconciliation Function (§4, stage 8) repairs the drift later.
